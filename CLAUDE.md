# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`obs-streamelements-core` is the C++ OBS Studio plugin that powers SE.Live (the StreamElements desktop UI/integration on top of OBS). The module name is `obs-streamelements-core` (built as an OBS module/shared library). On Windows it also builds a secondary executable `obs-streamelements-core-streamelements-restore-script-host` — an AngelScript runtime used by the backup-restore flow (see Backup section below).

It is **not** a standalone build — it expects to be checked out into an OBS Studio tree alongside `obs-browser`:

```
obs-studio/
  plugins/
    obs-browser/                    <- sibling repo, must exist
    obs-streamelements-core/        <- this repo (formerly built as plugins/obs-browser by CI)
  UI/obs-frontend-api/              <- include path picked up via ${CMAKE_SOURCE_DIR}
```

`StreamElementsGlobalStateManager.hpp` and `StreamElementsBrowserWidget.cpp` `#include "../../obs-browser/panel/browser-panel.hpp"` directly, so the obs-browser repo is a hard build dependency for headers. **Runtime CEF**, however, lives entirely inside the obs-browser process — this plugin talks to obs-browser CEF panels over a localhost websocket (see "Wire protocol" below). The `deps/cef-stub/` directory implements just enough of the CEF type system (`CefRefPtr` = `std::shared_ptr`, `CefValue`/`CefDictionaryValue`/`CefListValue` as in-memory trees, `CefParseJSON`/`CefWriteJSON` via `nlohmann::json`) for the plugin to use CEF-shaped data as its wire format without linking real CEF.

## Build, format, CI

- **Build system:** CMake. The project auto-detects whether it is being built standalone (`BUILD_OUT_OF_TREE=ON`) or from inside an obs-studio CMake project (`CMAKE_PROJECT_NAME STREQUAL "obs-studio"`). Out-of-tree builds require OBS's `find_qt` / `install_obs_plugin_with_data` CMake helpers to be available from the parent project's module path.
- **Qt:** `QT_VERSION` cache var, `5` or `6`. `AUTOMOC`, `AUTOUIC`, `AUTORCC` all on. Add new `.ui` files / Qt classes by listing them in `obs-streamelements-core_HEADERS`/`_SOURCES` in `CMakeLists.txt`; resources are bundled via `streamelements/streamelements.qrc` (see the `qt6_add_resources` block at the bottom of the CMakeLists when `QT_VERSION=6`).
- **Version stamp:** `streamelements/Version.generated.hpp` is generated at build time by CI (`printf "#define STREAMELEMENTS_PLUGIN_VERSION %s%06dL\n" $(date +%Y%m%d) ${CIRCLE_BUILD_NUM}`). It is required to compile but not checked in. Locally, create one with any sentinel value or run the CI command. `streamelements/Version.hpp` also declares `HOST_API_VERSION_MAJOR`/`MINOR` (currently 6.7) — bump major on breaking host-API changes.
- **Formatting:** clang-format with `.clang-format` at the repo root (Cpp11, `AccessModifierOffset: -8`, tabs). Run `./formatcode.sh` to format the whole tree; `CI/check-format.sh` fails CI if `git ls-files --modified` is non-empty after a format pass. There is no separate lint/test target — formatting compliance is the only CI gate.
- **CI:** `.circleci/config.yml` builds on macOS (xcode 13.1) by cloning `obs-studio`, replacing `plugins/obs-browser` with this repo, generating `Version.generated.hpp`, and running `./CI/full-build-macos.sh -b` from the OBS root. There is no test suite in this repo.

## Platform-specific bits worth knowing

- **Windows-only:** BugSplat crash reporting (`ENABLE_BUGSPLAT`), AngelScript-based `restore-script-host` executable (`ENABLE_ANGELSCRIPT`), `/MD`→`/MT` static-runtime swap, `/WX`→`/W1` warning downgrade, named-pipe `ControllerServer` (see API section). Crash handling on Windows lives in `StreamElementsCrashHandler.cpp` (uses `deps/StackWalker`).
- **macOS-only:** `StreamElementsCrashHandler.mm` — Mach exception port handling. `mach_excServer.c`/`mach_excUser.c` are generated by `mig` from `mach_exc.defs` at configure time (see the `add_custom_command` in `CMakeLists.txt`). BugSplat is loaded as an embedded framework from `deps/`. Build uses Xcode generator with dSYM emission configured per-variant.
- **Linux:** Builds (`INSTALL_RPATH "$ORIGIN/"`) but has no CI; treat changes there as best-effort.

## Architecture: the big picture

Plugin lifecycle (driven from `obs-streamelements-core-plugin.cpp`):

1. `obs_module_load` registers `audio_wrapper_source` (a passthrough source in `streamelements/audio-wrapper-source.{c,h}`).
2. `obs_module_post_load` registers a frontend event callback.
3. On `OBS_FRONTEND_EVENT_FINISHED_LOADING`, `StreamElementsGlobalStateManager::GetInstance()->Initialize(main_window)` constructs the entire subsystem graph.
4. On `OBS_FRONTEND_EVENT_EXIT` / `SCRIPTING_SHUTDOWN`, the global state manager is destroyed.
5. `obs_module_unload` runs `log_remaining_objects()` to log leaked OBS sources/outputs/encoders/services, then `SETRACE_DUMP()`.

### `StreamElementsGlobalStateManager` (the root)

`streamelements/StreamElementsGlobalStateManager.{hpp,cpp}` is the central singleton. It owns `std::shared_ptr`s to every manager subsystem; it is also the first place to look when tracing who owns what. Major managers (see `GlobalStateManager.hpp` for the full list):

- **UI / browser embedding:** `BrowserWidgetManager` (docked CEF panels), `BrowserDialog` (modal/non-modal CEF dialogs), `MenuManager`, `NativeOBSControlsManager` (overlays/replaces native OBS UI), `PreviewManager` (event filter on the preview pane), `ScenesListWidgetManager`, `SceneItemsMonitor`.
- **OBS state mirroring:** `ObsSceneManager`, `ObsActiveSceneTracker`, `ProfilesManager`, `BackupManager`, `CleanupManager`, `ObsAppMonitor` (base class; `GlobalStateManager` inherits from it).
- **Outputs / compositions** — the multistreaming engine; see the dedicated section below.
- **Networking / messaging:** `HttpClient` (CURL), `LocalFilesystemHttpServer` (loopback HTTP server, see "Local HTTP" below), `HttpServerManager`, `WebsocketApiServer` (`deps/websocketpp` + `asio`), `MessageBus`, `ControllerServer` (Windows named pipe, see below), `BandwidthTestManager`/`Client` + `ObsBandwidthTestClient`.
- **Misc:** `HotkeyManager`, `WorkerManager`, `AsyncTaskQueue`, `PerformanceHistoryTracker`, `AnalyticsEventsManager`, `RemoteIconLoader`, `CrashHandler`, `DeferredExecutive`.

When adding a new subsystem, the pattern is: write `Foo.hpp`/`Foo.cpp`, add to `obs-streamelements-core_SOURCES`/`_HEADERS` in `CMakeLists.txt`, add a `std::shared_ptr<Foo>` member + `GetFoo()` getter to `StreamElementsGlobalStateManager`, and initialize/destroy in `Initialize()` / dtor.

## Compositions (video and audio)

The plugin's central abstraction for "what gets rendered and encoded." Located in `streamelements/StreamElements{Video,Audio}Composition{,Manager}.{hpp,cpp}` and `StreamElementsSharedVideoCompositionManager.{hpp,cpp}`.

**Video composition class hierarchy** (`StreamElementsVideoComposition.hpp`):
- `StreamElementsVideoCompositionBase` — abstract base (`GetCurrentScene()`, `GetAllScenesInternal()`, `SerializeComposition()`, `AddScene()`, `SetCurrentScene()`, `RemoveScene()`, plus the reference-counted inner `CompositionInfo`).
- `StreamElementsObsNativeVideoComposition` — wraps the main OBS canvas. Singleton, non-removable (`CanRemove() == false`).
- `StreamElementsCustomVideoComposition` — fully independent custom composition, backed by an OBS **private canvas** (`obs_canvas_create_private`, scenes created with `obs_canvas_scene_create`). Owns its own `m_canvas`, scenes, video info, and encoder providers.
- `StreamElementsObsNativeVideoCompositionWithCustomEncoders` — hybrid: renders to the main OBS canvas but uses plugin-managed encoders. Lets the user multistream the main canvas with different encoder configs.

**Audio composition** mirrors this with `StreamElementsObsNativeAudioComposition` and `StreamElementsCustomAudioComposition`. Audio encoders are NOT lazy — each composition holds fixed arrays `obs_encoder_t* m_streamingAudioEncoders[MAX_AUDIO_MIXES]` and `m_recordingAudioEncoders[MAX_AUDIO_MIXES]` (see `StreamElementsAudioComposition.cpp:289-290`). Audio and video compositions have **independent lifecycles** and are bound 1:1 only at output-creation time — one video composition can be paired with multiple audio compositions and vice versa.

**Lazy video encoder creation** (recent direction, commits `04879d1`, `f0ac045`):

Encoders are NOT created at composition-creation time. The hierarchy (defined in `streamelements/StreamElementsUtils.hpp`):

- `SELazyOBSVideoEncoderAllocatorBase` — strategy interface for "how to allocate an encoder."
- `SELazyOBSVideoEncoderProvider` — holds an allocator and a consumer refcount. First call to `AddConsumer()` triggers `allocator->AllocRef()`; the count dropping to zero calls `allocator->Reset()` and releases the underlying `obs_encoder_t`.
- `SELazyOBSVideoEncoderProvider::SELazyOBSEncoderReference` — RAII handle returned to a consumer. Dtor calls `RemoveConsumer()`.

A composition exposes vectors of providers (`m_streamingVideoEncoders`, `m_recordingVideoEncoders`); outputs call `GetStreamingVideoEncoderRef(index)` / `GetRecordingVideoEncoderRef(index)` to obtain refs. **Multiple outputs pointing at the same encoder index share the same `obs_encoder_t`** — this is how multistreaming avoids re-encoding the same frames twice.

**Manager split:**
- `StreamElementsVideoCompositionManager` — registry of all compositions (native + custom) keyed by ID. Indexed lookup by scene, scene id, scene name, scene-item id, scene-item name. Handles `Deserialize…`/`Serialize…` round-trips for the API.
- `StreamElementsSharedVideoCompositionManager` — higher-level: tracks "shared" compositions that are exposed across host-API clients (connect/disconnect, lifecycle events). Driven by API calls `addSharedVideoComposition`, `connectVideoCompositionToSharedVideoComposition`, `disconnectVideoCompositionsFromSharedVideoCompositionsByIds`.

**Canvas-* headers** are NOT core rendering; they implement the UI editing gizmo for compositions (selection, transform handles, hit-testing). `canvas-controls.hpp` (~2k lines) is the bulk of the gizmo logic; `canvas-draw.hpp`, `canvas-scan.hpp`, `canvas-math.hpp`, `canvas-mutate.hpp`, `canvas-config.hpp` are supporting utilities. `StreamElementsVideoCompositionViewWidget` is the QWidget that hosts the preview and gizmo overlay; rendering itself is done by OBS on its render thread.

## Outputs / multistreaming

`OBS does not natively multistream.` The plugin adds it by managing N parallel `obs_output_t` instances bound to compositions. Located in `streamelements/StreamElementsOutput{,Manager,SettingsManager}.{hpp,cpp}`.

**Type enum** (`StreamElementsOutput.hpp:10`):
```cpp
enum ObsOutputType { StreamingOutput, RecordingOutput, ReplayBufferOutput };
```

**Class hierarchy** (`StreamElementsOutput.hpp`):
- `StreamElementsOutputBase` — common base.
- Streaming: `StreamElementsCustomStreamingOutput` (plugin-managed RTMP/SRT/etc., owns its own `obs_service_t m_service`), `StreamElementsObsNativeStreamingOutput` (wraps the OBS main stream; non-removable, `IsObsNativeOutput()` true).
- Recording: `StreamElementsCustomRecordingOutput`, `StreamElementsObsNativeRecordingOutput`.
- ReplayBuffer: `StreamElementsCustomReplayBufferOutput`, `StreamElementsObsNativeReplayBufferOutput`.

Each native output is a singleton inside the manager keyed `id="default"`; the plugin tracks but cannot remove or disable it (it follows OBS's `OBS_FRONTEND_EVENT_STREAMING_STARTED`/`STOPPED` etc.). Custom outputs are user-managed and removable.

**Manager state** (`StreamElementsOutputManager`):
```cpp
std::map<ObsOutputType,
         std::shared_ptr<std::map<std::string,
                                  std::shared_ptr<StreamElementsOutputBase>>>> m_map;
```
i.e. type → (id → output). The same map structure backs `SerializeAllOutputs(type)`, `DeserializeOutput(type, …)`, `RemoveOutputsByIds(type, …)`, `EnableOutputsByIds(type, …)`, `DisableOutputsByIds(type, …)`.

**Start flow** for a custom streaming output:
1. JS calls the host API (e.g. `addStreamingOutput`) — `OutputManager::DeserializeOutput()` instantiates `StreamElementsCustomStreamingOutput::Create()` and stashes it in `m_map`.
2. Output is `SetEnabled(true)` → `Start()` → `StartInternal()` which:
   - `obs_output_create(output_type_id, ...)` (e.g. `"rtmp_output"`).
   - Acquires lazy encoder refs from the bound video composition; first encoder via `obs_output_set_video_encoder(...)`, remaining via `obs_output_set_video_encoder2(m_output, encoder, i)` for **multi-track / multi-codec outputs**.
   - Audio encoders fetched per track from the audio composition and bound with `obs_output_set_audio_encoder(...)`.
   - `obs_output_set_service(m_output, m_service)` wires destination URL/key.
   - `obs_output_start(m_output)`.
   - Hooks OBS signal handlers for `start`, `stop`, `reconnect`, `reconnect_success`, etc.; each maps to a host-API event dispatched via the websocket bus.
3. Reconnect/retry is OBS's responsibility (configured by the encoder/service settings). Errors are surfaced as `error.message` and `stats.lastErrorMessage` fields in the serialized output state.

**Stop flow:** `obs_output_force_stop(m_output)` → release encoder ref handles (which decrements their consumer counts and may free the underlying `obs_encoder_t`) → `obs_output_release(SETRACE_DECREF(m_output))`. The manager's `Reset()` (called from its dtor) erases only non-native outputs.

**CefValue shapes** — the canonical JS-facing payload for a streaming output looks roughly like:
```jsonc
{ "id": "...",                              // optional; auto-generated
  "name": "...",
  "videoCompositionId": "...",             // omit / null = native
  "audioCompositionId": "...",
  "audioTracks": [0],
  "videoEncoders": [0, {...encoder spec...}],   // indices into composition encoders OR full encoder dicts
  "streamingSettings": {                   // becomes obs_service_t settings
    "type": "rtmp_custom",
    "serverUrl": "rtmp://...",
    "streamKey": "...",
    "useAuth": false, "authUsername": "", "authPassword": ""
  },
  "isEnabled": true }
```
Outputs serialize back with extra fields `isActive`, `canDisable`, `canRemove`, `isObsNativeOutput`, `error`, `stats` (`{ width, height, framesDropped, congestion, totalBytes, totalFrames, isPaused, isReconnecting, … }`).

## Host API and wire protocol

The plugin exposes ~200 host-API calls to in-process CEF panels and to obs-browser browser sources / dialogs / docking widgets. All routing goes through `StreamElementsApiMessageHandler` (`.cpp` is ~80KB — don't read top-to-bottom, search by handler name).

**Handler registration:** the file uses `API_HANDLER_BEGIN("name") … API_HANDLER_END()` macros (defined around line 490 of the .cpp) which expand to `RegisterIncomingApiCallHandler(name, lambda)` with a `s_sync_api_call_mutex` guard around the body. There is also an `API_HANDLER_END_ASYNC()` variant which omits `complete_callback()` so the handler can call it later. Handler names are **flat strings**, not dotted (e.g. `addStreamingOutput`, `enableStreamingOutputsByIds`, `getAllStreamingOutputs`, `setCurrentSceneById`, `triggerReplayBufferOutputSaveById`). The JS side may prefix with `host.*` but the registered IDs do not.

**Functional groups** (from grepping `API_HANDLER_BEGIN` names — ~204 total):
- Outputs: `addStreamingOutput`, `addRecordingOutput`, `addReplayBufferOutput`, `enableXOutputsByIds`, `disableXOutputsByIds`, `removeXOutputsByIds`, `getAllXOutputs`, `triggerRecordingOutputSplitById`, `triggerReplayBufferOutputSaveById`.
- Compositions: `addVideoComposition`, `addAudioComposition`, `addSharedVideoComposition`, `setVideoCompositionProperties`, `setVideoCompositionTransition`, `connectVideoCompositionToSharedVideoComposition`, `disconnectVideoCompositionsFromSharedVideoCompositionsByIds`.
- Encoders / settings: `getAvailableVideoEncoders`, `getAvailableAudioEncoders`, `getAvailableVideoEncoderClassProperties`, `getEncoderProperties`, `getEncodingSettings`, `setEncodingSettings`.
- Scenes & items: `getAllScenes`, `addScene`, `setCurrentSceneById`, `removeScenesByIds`, `addSceneItemBrowserSource`/`GameCaptureSource`/`VideoCaptureSource`/`ObsNativeSource`/`Group`, `setSceneItemPropertiesById`, `setSceneItemPositionInViewport`, `openSceneItemPropertiesDialogById`, etc.
- Scene collections / profiles: `getAllSceneCollections`, `addSceneCollection`, `setCurrentSceneCollectionById`, `getAllProfiles`, `setCurrentProfile`.
- Docking & dialogs: `addDockingWidget`, `showModalDialog`, `showNonModalDialog`, `setDockingWidgetUrlById`, `showStatusBarTemporaryMessage`, `showNotificationBar`.
- Backup/restore: `createUserEnvironmentBackupPackage`, `queryUserEnvironmentBackupPackageContent`, `restoreUserEnvironmentBackupPackageContent`.
- System / misc: `getHostProperties`, `getSystemHardwareProperties`, `getSystemCPUUsageTimes`, `getSystemMemoryUsage`, `takeScreenshot`, `httpRequestText`, `crashProgram`, `deadlockProgram` (debug helpers).

**Wire protocol — `WebsocketApiServer`** (`streamelements/StreamElementsWebsocketApiServer.{hpp,cpp}`):
- Plain JSON over websocket; default port `27952` (`m_port = 27952` in the header). Loopback only; no auth — security is process boundary.
- Clients open `ws://127.0.0.1:27952`, send a `register` message with a target type (e.g. `dialogHost`, `browserSource`, `dockingWidget`), receive a `register:response` with a per-connection `unique_id`.
- Subsequent messages are `{ "type": "dispatch", "payload": { "name": "MSG_INCOMING_API_CALL", "args": [...] } }` — the `args` carry the handler name and `CefListValue`-shaped argument list. Decoded via `nlohmann::json` (in `cef-stub`) into `CefValue` trees and dispatched to the registered handler.
- Server-to-client events use `DispatchJSEvent(type, target, event, eventArgsJson)`. Per-client `DispatchClientMessage()` does the actual send. Broadcast is implemented by iterating the target → connection-handle map in the server.

**Threading:** the websocket server runs its own thread (asio event loop). When a request arrives, `StreamElementsApiMessageHandler::OnProcessMessageReceived` queues a lambda onto the Qt main thread via `QtPostTask` (macro in `StreamElementsUtils.hpp:192`, implemented with `QMetaObject::invokeMethod(qApp, ..., Qt::QueuedConnection)` in `StreamElementsUtils.cpp:290`). All OBS-touching code therefore runs on the Qt/main UI thread; results are sent back on the asio thread when the lambda calls `complete_callback`.

**Local HTTP** (`StreamElementsLocalFilesystemHttpServer`) — loopback HTTP server that the plugin uses to serve local plugin assets (HTML/JS bundled in the qrc, or files on disk) to CEF panels, working around the fact that CEF browser-source pages cannot reliably load `file://` URLs. `StreamElementsHttpServerManager` exposes a configurable variant to host-API clients (`addBrowserScopedHttpServer`).

**ControllerServer** (`StreamElementsControllerServer.{hpp,cpp}`) — Windows-only named-pipe server (`\\.\pipe\StreamElementsObsLiveControllerServer`) used by external controllers such as Elgato Stream Deck integrations. JSON message format with `source`, `target` (`scope: "broadcast"` allowed), `payload`. Distinct from the websocket bus and used only for out-of-process device control.

**MessageBus** (`StreamElementsMessageBus.{hpp,cpp}`) — in-process pub/sub that fans events out to all connected CEF clients via `destination_filter_flags_t`. The `BrowserWidget` records its destination flags so the bus knows which surfaces should receive which events.

## Scene mirroring, profiles, backup

- **`StreamElementsObsSceneManager`** — top-level scene/source serialization/deserialization (`SerializeObsScenes`, `DeserializeObsScene`, `SetCurrentObsSceneById`, `SerializeObsSceneItems`). Owns the `SceneItemsMonitor` and `ScenesListWidgetManager`. Hooks OBS frontend events.
- **`StreamElementsObsActiveSceneTracker`** — narrower scope: tracks the *current transition source* and re-binds signal handlers when scene-collection / profile / transition change. Event-driven via `obs_frontend_add_event_callback`. Dispatches `DispatchTransitionChanged()`.
- **`StreamElementsSceneItemsMonitor`** — manages the items dock (sources list); stores per-scene-item auxiliary metadata (icons, default actions, context menus — some are feature-flagged off) and throttles UI refresh via `StreamElementsDeferredExecutive`. Updates are pulled via `Update()` from the scene manager, not pushed.
- **`StreamElementsScenesListWidgetManager`** — decorates (does not replace) the OBS scenes dock; maintains a custom toolbar, handles scene rename/double-click, similar auxiliary-metadata layer.
- **`StreamElementsPreviewManager`** — only installs a Qt event filter to forward preview-pane mouse events (double-click, focus changes) as host-API events. It does **not** render or manage the preview itself.
- **`StreamElementsProfilesManager`** — thin 1:1 wrapper over `obs_frontend_*_profile` APIs. Blocks profile switches while streaming/recording. Profiles live in OBS's standard config tree.
- **`StreamElementsBackupManager`** — zip-based backups of scene collections + profiles + sources + auxiliary metadata. File references in scene JSON are normalised to `<streamelements:relative-path:obs-studio>` monikers for portability. Restore extracts the zip, rewrites monikers back to absolute paths, and (Windows only, when `ENABLE_ANGELSCRIPT` is on) spawns `obs-streamelements-core-streamelements-restore-script-host.exe` to run AngelScript-driven post-restore logic outside the OBS process. Temp paths are registered with `CleanupManager::AddPath()` for shutdown cleanup.
- **`StreamElementsCleanupManager`** — registry of paths to delete at shutdown (`AddPath` / `Clean`). Used by backup, dialogs, etc.
- **`StreamElementsConfig`** — INI file `obs-streamelements-core.ini` in `obs_module_config_path("")` (OBS's standard plugin config dir, platform-dependent: `%APPDATA%\obs-studio\plugin_config\…` on Windows, `~/.config/obs-studio/plugin_config/…` elsewhere). Read/written via OBS's `config_t` API. Sections include `[Header]` (`Version`) and `[Startup]` (`Flags` bitfield with `ONBOARDING_MODE`, `SIGNED_IN`; `State`; aux menu/source/scene action JSON blobs). Also exposes `Read/WriteScopedJsonFile` for arbitrary scoped JSON storage in the plugin's config subtree.

## Conventions and gotchas

- **Tracing:** `SETrace.{hpp,cpp}` provides a custom trace facility; `SETRACE_DECREF`/`SETRACE_INCREF`/`SETRACE_ADDREF`/`SETRACE_NOREF` macros wrap OBS reference-counted-object ops (e.g. `obs_output_release(SETRACE_DECREF(m_output))`, `obs_scene_release(SETRACE_DECREF(scene))`). Preserve these when refactoring; on shutdown they're dumped by `SETRACE_DUMP()`.
- **Async-call markers:** `SEAsyncCallContextMarker asyncMarker(__FILE__, __LINE__);` is placed at the top of async entry points (frontend event callbacks, websocket dispatch). Keep them.
- **Threading:** Lots of `std::shared_mutex` / `std::recursive_mutex`. Several subsystems use `deps/moodycamel/{blocking,}concurrentqueue.h`. **All OBS API calls must run on the Qt main thread** — when in doubt, wrap with `QtPostTask(…)`. The websocket dispatch path already does this for handlers; new entry points that originate off-thread (e.g. signal handlers, asio callbacks) usually need to.
- **OBS object lifetime:** Encoders/outputs/scenes/sources/services are reference-counted. The `log_remaining_objects()` shutdown sweep will warn about leaks — if you see those warnings appear, you forgot a `release` / `SETRACE_DECREF`. Lazy-encoder consumer refs (`SELazyOBSEncoderReference`) similarly need to fall out of scope on stop.
- **CefValue is the wire format, not a real CEF object.** `CefRefPtr<CefValue>`/`CefDictionaryValue`/`CefListValue` are in-process trees in `deps/cef-stub/`. JSON in/out goes through `CefParseJSON`/`CefWriteJSON` which use `nlohmann::json`. Don't reach for real CEF APIs — many are stubbed and will silently no-op.
- **`Serialize…`/`Deserialize…` naming.** `SerializeXxx(output)` writes plugin state into a `CefValue`; `DeserializeXxx(input, output)` reads one in. Follow this convention for new API surface — it's how everything is plumbed through `ApiMessageHandler`.
- **No tests.** Don't claim a change is verified by tests; verify by building inside an OBS tree and exercising the plugin.
- **Generated/vendored files at the repo root to ignore:** `qrc_streamelements.cpp` (1.7 MB AUTORCC output checked in), `mach_excServer.c`/`mach_excUser.c` (regenerated by `mig`), and the `obs-streamelements-core*_autogen/` directories (Qt MOC output). Don't edit them.
- **Don't add new top-level docs files** unless explicitly requested; the existing `README.md` is intentionally one line.
